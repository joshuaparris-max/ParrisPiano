The North Star

The “shooter” bit is just the game skin on top of that loop.

Core features (what “working” looks like)

* Low-latency input capture (notes, sustain pedal, velocity)

Definition of done: press keys → app sees it instantly; app plays MIDI out → piano sounds it.

The “Shooter Teacher” gameplay layer (the fun bit)
2) Visual note world (the targets)
You need a rendering system that can show:
* Note lanes (like a highway) OR a “shooting gallery” where notes become targets
* Incoming notes scrolling to a “hit line”
* Chords shown as stacks / clusters
* Left-hand vs right-hand lanes

Definition of done: load a MIDI → it becomes visible targets with timing.
3) The shooter mechanics
A “shooter” variant that maps to piano skills:
* Notes = targets
* You “shoot” by playing the right note at the right time
* Streaks, combos, multipliers
* Boss moments = big chords / runs
* Power-ups = perfect timing streak, “slow time”, “hint burst”
* Miss penalties that teach (not punish): slow down, isolate bar, simplify part
Definition of done: playing correctly feels like winning a game.

The “teacher brain” (what makes it personalised, not just a game)
4) Timing engine + musical alignment
This is where most apps fail unless done properly.
* Convert MIDI to a timeline (ticks → seconds) with tempo changes handled
* Support tempo scaling (practice slower)
* Keep a playhead that can “wait” (guide mode) or run freely (performance mode)
Definition of done: what you play can be compared to what the song expects.
5) Scoring engine (accurate + fair)
You need scoring that understands music, not just “hit/miss”:
* Pitch accuracy (right notes)
* Rhythm accuracy (early/late windows)
* Chord handling (all notes within a window count as “together”)
* Sustain pedal correctness (optional)
* Dynamics/velocity (optional)
* Finger independence metrics (patterns like repeated-note stability)
Outputs:
* Per-note judgement (Perfect/Good/Late/Early/Miss/Wrong)
* Per-bar score and heatmap
* Mistake clusters (e.g., always late on syncopation)
Definition of done: after you play, it can tell you exactly what went wrong.
6) Diagnosis engine (the “why”)
This turns raw mistakes into coachable insights:
* “Your left hand is consistently late by ~80ms.”
* “You’re dropping the 3rd note of chords.”
* “You rush after rests.”
* “You lose timing when the melody jumps.”
This can start rule-based, then become AI-assisted.
Definition of done: feedback feels specific and true.
7) Personalisation model (memory + adaptive difficulty)
You need a per-user profile with:
* Skill ratings: timing, sight-reading, chord fluency, hand independence, etc.
* Repertoire history and what’s mastered
* Weakness tags (auto-generated)
* Preferred practice length, motivation style, kid mode, etc.
Then an algorithm that chooses:
* What to practise next
* At what tempo
* How many reps
* When to revisit (spaced repetition)
Definition of done: two different people get different practice plans from the same song.
________________


The “AI coach” layer (chat + explanations + planning)
8) Coaching chat that’s grounded in your performance
This is the key: the AI shouldn’t waffle — it should reference your data. So you need:
* A structured “session summary” object (JSON) the AI reads
* A set of “coach tools” the AI can call:
   * suggest_exercises()
   * generate_practice_plan()
   * explain_rhythm() with visual examples
   * make_kid_friendly_story_prompt() (optional)
Definition of done: “You missed F#4 in bars 12–14” not “keep practising!”
9) Content generation / adaptation (optional but huge)
To be personalised, the app eventually needs to create variations:
* Simplify: melody-only, block chords, reduced LH, easier rhythm
* Loop tricky bars automatically
* Generate drills (e.g., chord transitions, scale fragments)
* Auto-fingerings (hard but doable with heuristics)
Definition of done: it can turn “Let It Go” into “Let It Go (easy)” on the fly.
